Current State

The managed bundle is the real runtime boundary.

The repository checkout and the installed OpenClaw bundle are not the same thing after install. The installer copies skills, scripts, cron files, and docs into a managed bundle directory, registers that bundle in openclaw.json, and then the running skills execute relative to that installed path.

Runtime surface Owner Why it matters in this review
scripts/install_bundle.py Repo checkout Defines how the bundle is copied, where runtime directories are created, and how openclaw.json is updated.
~/.openclaw/bundles/compound-clawskill Managed install This is the actual runtime boundary for installed skills. Any onboarding path that writes elsewhere can mislead the user.
openclaw.json.skills.load.extraDirs[] OpenClaw config Registers the installed skills directory so OpenClaw can discover the bundle.
skills/*/SKILL.md with {baseDir} Installed skill runtime These commands resolve {baseDir}/../../longevityOS-data, which points into the installed bundle.
Intent and Gap

The old docs described a path that could make the install look seeded when it was not.

The problem was not that seed files were missing. The problem was that the docs told the user to seed the repo checkout, while the installed skills read the managed bundle. That split creates a false-success state: data exists somewhere, but not where the running bundle looks.

Question Old answer New answer
Where does exploration data go? Implicitly into repo-local longevityOS-data Explicitly into ~/.openclaw/bundles/compound-clawskill/longevityOS-data
Who owns the seed path? User copies files by hand Installer owns the path via python3 scripts/install_bundle.py --seed-data
What happens if real data already exists? No guard in the docs path Seeded install fails loudly on a non-empty installed data root
What changed in runtime schemas? Not applicable No nutrition, health, insights, or news schema change
Direction

The fix stays inside setup, avoids runtime churn, and leaves future cron automation as a separate decision.

This PR is deliberately scoped. It fixes the install path, clarifies the contract, and restores green tests. It does not change nutrition logging, health profile structure, or the runtime behavior of the user-facing skills.

What improves immediately

  • Seeded installs and runtime reads now target the same tree.
  • Users get an explicit command instead of a path they have to infer.
  • Contributors see a green full suite in the normal checkout.

What still stays manual

  • Cron JSON still needs a Telegram DM chat id.
  • Cron JSON still needs a local timezone review.
  • That work is intentionally held for a separate decision.

Why this stays safe

  • The seeded path fails on a non-empty installed data root.
  • No user data files are migrated or reshaped.
  • Verification still uses the live OpenClaw CLI readiness check.
